# v1.0.0-lab: notas de versión y runbook de release/rollback

## Estado

Este documento prepara una futura versión denominada `v1.0.0-lab`. La etiqueta y
la GitHub Release todavía no existen, y este runbook todavía no ha sido
ejercitado.

La candidata es una **validated laboratory reference architecture** y está
**not production-ready**. Su publicación no autoriza usuarios o datos reales, un
entorno persistente ni una declaración de preparación para producción.

## Objetivo y alcance

`v1.0.0-lab` identifica un punto reproducible de la arquitectura de referencia
validada en laboratorio. Incluye aplicación, Infrastructure as Code,
documentación, tests, workflows y guardrails ya versionados en el repositorio.

La release debe permitir relacionar de forma verificable:

1. El commit candidato.
2. La etiqueta `v1.0.0-lab`.
3. La GitHub Release.
4. El SBOM SPDX JSON generado para ese mismo commit.
5. El SHA-256 publicado del SBOM.

## Controles y evidencias incluidos

- API Gateway con Cognito, scopes JWT y CORS explícito.
- Acceso DynamoDB Query, métricas incrementales y paginación por cursor.
- EventBridge, SQS, DLQ, reintentos e idempotencia.
- CloudWatch dashboard, alarmas y runbooks.
- GitHub Actions con OIDC y credenciales STS temporales.
- Despliegues de laboratorio efímeros con destrucción y verificación.
- Baselines controlados local y AWS para WA-017.
- Entrega real `ALARM` y `OK` a Slack mediante SNS y Amazon Q Developer para
  WA-014.
- AWS Budgets y Cost Anomaly Detection evidenciados para la cuenta de laboratorio.
- Validate, CodeQL, Dependabot, guardrail de secretos y workflow SBOM.

Las evidencias históricas permanecen en sus documentos originales. No deben
editarse ni reinterpretarse como pruebas de capacidad u operación de producción.

## Prerrequisitos de release

Antes de crear la etiqueta:

1. Trabajar desde el checkout autoritativo en `mirofish`.
2. Confirmar que GitHub es la fuente de verdad y que `main` coincide con
   `origin/main`.
3. Confirmar un working tree limpio y sin archivos sin seguimiento.
4. Registrar el commit candidato como `<CANDIDATE_COMMIT_SHA>` únicamente cuando
   exista la decisión real de release.
5. Verificar que Validate y CodeQL están completados correctamente para ese SHA.
6. Ejecutar todos los tests y guardrails locales aplicables.
7. Revisar vulnerabilidades abiertas y documentar su corrección o aceptación.
8. Confirmar que las notas no contienen secretos, identificadores sensibles ni
   resultados inventados.
9. Confirmar que no existe ya una etiqueta o release `v1.0.0-lab` conflictiva.
10. Obtener aprobación explícita para crear el tag y la GitHub Release.

## Comprobación del commit candidato

Durante la release real, sustituir los marcadores solo con valores observados:

```bash
git fetch origin
git switch main
git merge --ff-only origin/main
git status --short --branch
git rev-parse HEAD
git rev-parse origin/main
```

Registrar:

```text
CANDIDATE_COMMIT_SHA=<CANDIDATE_COMMIT_SHA>
```

Los dos `rev-parse` deben coincidir exactamente con
`<CANDIDATE_COMMIT_SHA>`. Si no coinciden o el árbol no está limpio, abortar.

## Procedimiento futuro para crear la etiqueta

Estas instrucciones son para una ejecución futura y aprobada; no crean la
etiqueta durante la preparación documental:

```bash
git tag -a v1.0.0-lab <CANDIDATE_COMMIT_SHA> \
  -m "v1.0.0-lab validated laboratory reference architecture"
git show --no-patch --decorate v1.0.0-lab
git push origin v1.0.0-lab
```

Después del push, verificar que la referencia remota apunta exactamente al objeto
esperado. No mover silenciosamente una etiqueta publicada.

## Procedimiento futuro para crear y verificar la GitHub Release

Preparar las notas finales desde este documento, conservando la declaración
`not production-ready`, las limitaciones y el backlog abierto. Cuando el SBOM y su
hash estén disponibles, una ejecución aprobada podrá usar:

```bash
gh release create v1.0.0-lab \
  --repo fermarfer1982/aws-cloudops-incident-hub \
  --title "v1.0.0-lab" \
  --notes-file <FINAL_RELEASE_NOTES_PATH> \
  <SBOM_PATH> \
  <SBOM_SHA256_PATH>
```

Verificar después:

```bash
gh release view v1.0.0-lab \
  --repo fermarfer1982/aws-cloudops-incident-hub
```

Registrar durante la ejecución real:

```text
RELEASE_URL=<OBSERVED_RELEASE_URL>
```

No inventar ni anticipar esa URL en el repositorio.

## SBOM SPDX JSON y asociación verificable

El SBOM debe generarse desde el checkout exacto de `<CANDIDATE_COMMIT_SHA>` con el
workflow versionado `.github/workflows/sbom.yml` o un mecanismo aprobado que
produzca el mismo formato SPDX JSON. No se debe reutilizar un artifact de otro
commit.

La ejecución real debe verificar y registrar:

```text
SBOM_COMMIT=<CANDIDATE_COMMIT_SHA>
SBOM_TAG=v1.0.0-lab
SBOM_RELEASE=<OBSERVED_RELEASE_URL>
SBOM_SHA256=<OBSERVED_SBOM_SHA256>
```

Calcular el hash desde el archivo finalmente adjuntado:

```bash
sha256sum <SBOM_PATH> | tee <SBOM_SHA256_PATH>
sha256sum --check <SBOM_SHA256_PATH>
```

El nombre, tamaño y SHA-256 observados deben publicarse en las notas de release.
WA-022 continúa parcial hasta que el SBOM, el hash, el commit, el tag y la release
puedan relacionarse y verificarse.

## Verificación posterior a la release

1. La etiqueta remota resuelve a `<CANDIDATE_COMMIT_SHA>`.
2. La GitHub Release usa `v1.0.0-lab` y no aparece como producción.
3. Las notas declaran `validated laboratory reference architecture` y
   `not production-ready`.
4. El SBOM y su archivo SHA-256 están adjuntos.
5. La comprobación SHA-256 pasa sobre el artifact descargado de la release.
6. Validate y CodeQL siguen aprobados para el commit etiquetado.
7. Los enlaces de las notas y evidencias son válidos.
8. No se ha iniciado ningún despliegue AWS como efecto del proceso de release.

Registrar únicamente resultados observados. La existencia de este runbook no es
evidencia de que estas comprobaciones hayan sido ejecutadas.

## Criterios de aborto

Abortar antes de publicar si:

- `main`, `origin/main` y el commit candidato no coinciden.
- El working tree no está limpio.
- Validate, CodeQL, tests o guardrails fallan o siguen pendientes.
- Existe una vulnerabilidad sin decisión documentada incompatible con la release.
- El SBOM no corresponde al commit candidato o no puede verificarse.
- Faltan las declaraciones de alcance de laboratorio y no producción.
- Aparecen secretos, identificadores sensibles o evidencia no verificable.
- La etiqueta o release ya existe con otro commit.
- Falta aprobación explícita para la acción remota.

## Rollback futuro del tag y de la GitHub Release

El rollback de publicación requiere aprobación explícita y un registro del motivo.
Primero conservar la evidencia necesaria y marcar la release como retirada o
incorrecta. Si la política aprobada exige eliminarla:

```bash
gh release delete v1.0.0-lab \
  --repo fermarfer1982/aws-cloudops-incident-hub \
  --yes
git push origin :refs/tags/v1.0.0-lab
git tag -d v1.0.0-lab
```

Verificar la ausencia en GitHub y en las referencias remotas. Registrar el
resultado real como `<OBSERVED_ROLLBACK_RESULT>`; este documento no afirma que el
rollback haya ocurrido.

No reutilizar `v1.0.0-lab` para otro commit sin una decisión explícita y
documentada. Cuando sea posible, publicar una nueva versión corregida conserva
mejor la trazabilidad que mover una etiqueta existente.

Eliminar una GitHub Release o una etiqueta **no revierte despliegues AWS**, no
elimina stacks y no modifica recursos cloud.

## Restricciones de despliegue

Este procedimiento no autoriza despliegues directos desde `mirofish`. Cualquier
despliegue AWS debe continuar exclusivamente mediante GitHub Actions, OIDC y
credenciales temporales, con aprobación, mínimo privilegio, destrucción del stack
efímero y verificación de limpieza.

La publicación de `v1.0.0-lab` es una operación de repositorio y release; no es una
autorización para ejecutar workflows AWS.
